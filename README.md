# AI Quiz App

A small full-stack quiz app where every question, hint, and piece of feedback is
generated by an LLM through the [OpenRouter](https://openrouter.ai) API.

- **Frontend:** React 18 + Vite
- **Backend:** Node.js + Express
- **AI:** OpenRouter (default model `openai/gpt-4o-mini`; swap to GPT-4, Claude, Llama, etc.)

---

## Features

- Pick any topic and the AI writes the multiple-choice quiz on the fly.
- Choose `easy` / `medium` / `hard` difficulty and 3–10 questions.
- "Get an AI hint" button gives a Socratic nudge without leaking the answer.
- Submitting the quiz returns per-question AI feedback alongside the correct answer.
- Answer key never reaches the browser — grading happens server-side.
- Responsive layout with keyboard-friendly controls and a `prefers-reduced-motion` fallback.

## Setup

You'll need Node.js 18+ (uses native `fetch` and `node --watch`).

### 1. Get an OpenRouter API key

Sign in at <https://openrouter.ai/keys> and create a key.

### 2. Backend

```bash
cd backend
npm install
cp .env.example .env        # then open .env and paste your key
npm run dev                 # http://localhost:3001
```

Environment variables (see `backend/.env.example`):

| Variable                | Default                 | Notes                                    |
| ----------------------- | ----------------------- | ---------------------------------------- |
| `OPENROUTER_API_KEY`    | _required_              | Your key from openrouter.ai              |
| `OPENROUTER_MODEL`      | `openai/gpt-4o-mini`    | Any OpenRouter model slug                |
| `OPENROUTER_SITE_URL`   | `http://localhost:5173` | Sent as the `HTTP-Referer` header        |
| `OPENROUTER_APP_NAME`   | `AI Quiz App`           | Sent as the `X-Title` header             |
| `PORT`                  | `3001`                  | Backend port                             |

### 3. Frontend

In a second terminal:

```bash
cd frontend
npm install
npm run dev                 # http://localhost:5173
```

Vite proxies `/api/*` to the backend, so no extra configuration is needed.

## API

All endpoints are JSON over HTTP. The frontend uses these via `src/api.js`.

### `GET /api/health`

Health check. Returns `{ ok: true, model }`.

### `POST /api/quiz`

Generate a new quiz.

Request:

```json
{ "topic": "The solar system", "numQuestions": 5, "difficulty": "medium" }
```

Response (note: `correctIndex` is NOT included):

```json
{
  "quizId": "uuid",
  "topic": "The solar system",
  "difficulty": "medium",
  "questions": [
    { "id": "uuid", "question": "…", "options": ["A","B","C","D"] }
  ]
}
```

### `POST /api/quiz/:quizId/hint`

```json
{ "questionId": "uuid" }
```

Returns `{ "hint": "…" }`.

### `POST /api/quiz/:quizId/submit`

```json
{ "answers": { "<questionId>": 2, "<questionId>": 0 } }
```

Returns the graded result plus per-question AI feedback:

```json
{
  "quizId": "uuid",
  "score": 4,
  "total": 5,
  "results": [
    {
      "id": "uuid",
      "question": "…",
      "options": ["…"],
      "selectedIndex": 2,
      "correctIndex": 1,
      "isCorrect": false,
      "feedback": "AI-generated explanation…"
    }
  ]
}
```

## Design choices

**Why OpenRouter.** OpenRouter exposes an OpenAI-compatible chat-completions
endpoint that fronts dozens of models (GPT-4, Claude, Llama, Mistral…). One
integration, one billing surface, easy to A/B different models by changing the
`OPENROUTER_MODEL` env var. Because the wire format is OpenAI-compatible, we
can drop the official OpenAI SDK in later with almost no code change.

**Server-side grading.** The full quiz (including the answer key and rationales)
lives only on the backend in an in-memory `Map` keyed by `quizId`. The
`publicQuiz()` projection strips `correctIndex` and `explanation` before
serializing to the client. This means a user can't open DevTools and read the
answers — they have to submit and let the server grade them.

**JSON mode + defensive parsing.** Quiz generation asks the model for strict
JSON via `response_format: { type: "json_object" }` and a tightly scoped schema
in the prompt. Some models still occasionally wrap output in ` ```json ` fences
or add stray prose, so `extractJson()` handles fences and falls back to slicing
the first `{…}`/`[…]` block. Every question is then validated: 4 options
exactly, valid `correctIndex` in `[0, 3]`, non-empty question text. Bad items
are dropped silently; if nothing survives we return a 502 so the user can retry.

**Three distinct prompts.**

- *Questions* use temperature `0.7` for variety.
- *Hints* use a Socratic system prompt that explicitly forbids quoting any
  option verbatim or revealing the correct choice, at temperature `0.6`.
- *Feedback* is a tutoring prompt at temperature `0.5` — lower because we
  want accuracy here, not creativity.

**Feedback in parallel with a fallback.** Submission fires N feedback calls
concurrently via `Promise.all`. Each call is wrapped so that if any single
feedback request fails, we fall back to the deterministic explanation the
model produced when it first wrote the question. The user is never left
staring at a blank result row.

**Frontend simplicity.** With three screens (start → quiz → results), a
single `App.jsx` view state machine is clearer than React Router. Vite's
`/api` proxy keeps the frontend code free of `localhost:3001` URLs and avoids
CORS friction in dev. No global state library — `useState` and prop drilling
are sufficient at this size and easier to read.

**Styling.** A small, hand-written CSS file with custom properties for the
palette, fluid type via `clamp()`, a single mobile breakpoint, and a
`prefers-reduced-motion` rule. No Tailwind / CSS-in-JS adds zero build
complexity and zero runtime overhead.

## Trade-offs

- **In-memory quiz store.** Simple and dependency-free, but doesn't survive a
  restart and won't work across multiple backend instances. Swap in Redis or
  Postgres for production. A 1-hour TTL is enforced via a periodic prune.
- **No auth or rate limiting.** Anyone who can reach the backend can spend
  your OpenRouter credits. Add an API key check, per-IP rate limiting, or
  a per-session quota before exposing this publicly.
- **One LLM call per feedback item.** For a 10-question quiz that's 10
  parallel requests on submit. Cheaper alternative: ask the model once for
  all feedback in a single batched call. We chose per-item to keep the
  prompt small/focused and to fail gracefully per question.
- **JSON-mode reliability varies by model.** Cheaper/smaller models
  occasionally produce malformed JSON. The parser is defensive but a truly
  hostile output will surface a 502 and require a retry. A future improvement
  is to retry once with `temperature: 0` and a stricter "fix the JSON" prompt.
- **No streaming.** Quizzes appear all at once after the generation request
  finishes. Streaming would feel snappier but complicates validation and the
  "hide answers from the client" guarantee.
- **No question deduplication or persistence.** Re-generating the same topic
  often yields overlapping questions across runs. A real product would
  persist generated questions and dedupe by embedding similarity.
- **Hint leakage risk.** The system prompt instructs the model not to reveal
  the answer, but LLMs aren't perfectly obedient. A safer (but more expensive)
  design would generate hints up-front with the question and have a second
  pass verify the hint doesn't contain any option text.
